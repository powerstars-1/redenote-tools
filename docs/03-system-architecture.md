# 系统架构

## 架构原则

服务端要尽量保持“轻、稳、可部署”。上游采集逻辑需要和传输层、校验层、响应标准化层解耦。

## 逻辑分层

### 1. API 层

职责：

- 暴露 HTTP 接口
- 进行输入校验
- 分配请求 ID
- 对敏感字段做日志脱敏
- 将内部异常映射为稳定的错误响应

### 2. 应用层

职责：

- 编排搜索与详情两个核心用例
- 将 API 参数翻译为 `Spider_XHS` 可接受的参数
- 对上游返回结果做标准化
- 应用分页与页数限制规则

### 3. 适配层

职责：

- 封装 `Spider_XHS` 调用
- 隔离上游代码变更
- 将上游异常转换为内部定义的类型化错误

### 4. 支撑层

职责：

- 结构化日志
- 超时与重试策略
- 配置加载
- 健康检查接口
- 为后续缓存、限流、配额预留扩展点

## 第一阶段部署形态

第一阶段采用单体可部署服务：

- 一个 HTTP 服务进程
- 一个内嵌的 `Spider_XHS` 适配层
- 暂不引入后台任务队列
- 暂不引入持久化账号存储

## 未来部署形态

当请求量增长后，可以逐步演进为：

- API 网关或统一入口层
- 无状态应用实例
- 分布式缓存
- 异步任务队列
- 平台维度的专用 worker

## 请求流程

### 搜索流程

1. 客户端提交搜索请求，包含关键词、筛选条件、页数和 Cookie。
2. API 层完成输入校验与日志脱敏。
3. 应用层将请求参数映射为 `Spider_XHS` 搜索参数。
4. 适配层从第 1 页开始按页调用 `Spider_XHS`。
5. 应用层聚合并标准化结果。
6. API 层返回统一 JSON 响应。

### 详情流程

1. 客户端提交笔记 URL 和可选 Cookie。
2. API 层校验 URL 格式。
3. 应用层调用 `Spider_XHS` 详情适配器。
4. 适配层获取详情并返回标准化结果。
5. API 层返回统一 JSON 响应。

## 安全与运维规则

- 不在数据库中持久化请求级 Cookie。
- 不在日志中输出 Cookie。
- 拒绝超大请求体。
- 对所有上游调用施加超时上限。
- 为缓存、限流、配额能力预留扩展点。
