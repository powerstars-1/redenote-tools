# 技术选型

## 已选基础项目

当前实现方向确定为：以 `Spider_XHS` 作为小红书内容检索能力的上游基础项目，进行服务端二次开发。

## 为什么选择 Spider_XHS

在当前第一阶段范围内，`Spider_XHS` 与需求的匹配度最高，因为它已经具备：

- 关键词搜索
- 搜索排序
- 图文/视频类型筛选
- 发布时间预设筛选：不限、一天内、一周内、半年内
- 按请求注入 Cookie

它也更适合当前的交互模型：由调用方按请求传入 Cookie，而不是依赖服务端托管共享账号。

## 为什么第一阶段不选 MediaCrawler

`MediaCrawler` 在浏览器会话管理和长期稳定性上更强，但对当前阶段来说有两个明显不足：

- 搜索筛选能力还需要额外改造
- 其服务模型更偏任务调度型，不是轻量同步的搜索/详情 API

后续如果项目进入更高稳定性和更复杂登录流程阶段，`MediaCrawler` 仍然值得作为参考，尤其适合借鉴：

- 浏览器会话持久化
- 登录编排
- 抗检测策略

## 为什么第一阶段不选 XHS-Downloader

`XHS-Downloader` 更适合以下场景：

- 按 URL 提取详情
- 提取下载地址
- 文件下载流程
- 借助浏览器用户脚本提取链接

它的服务接口核心是“作品详情与下载”，不是“服务端主动关键词搜索”，因此不适合作为当前搜索优先项目的主底座。

## 当前技术路线

第一阶段按以下路线推进：

1. 基于 `Spider_XHS` 封装一层干净的服务适配器。
2. 将上游字段标准化为稳定的服务响应模型。
3. Cookie 仅按请求使用，不做服务端持久化。
4. 从第一版起就加入日志规范、输入校验、超时控制和故障边界。

## 风险说明

虽然“调用方传入 Cookie”降低了服务端存储凭据的风险，但并没有消除平台风险和运维风险。服务仍然需要具备：

- 严格的请求体日志脱敏
- 按用户与全局维度的限流
- 超时控制与熔断能力
- 对失效 Cookie、异常 Cookie 的清晰错误处理
